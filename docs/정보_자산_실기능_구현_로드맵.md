# 자산 실기능 구현 로드맵 (시뮬레이션 → 실제 동작)

> **목적**: 어드밴스드 에디터 자산 중 현재 시뮬레이션/클라이언트만 동작하는 약 90%를, 실제 저장·실시간·API 연동이 되도록 단계적으로 구현한다.  
> **원칙**: 유저가 “자신만의 웹/앱”을 만들고 배포하므로, **모든 데이터는 페이지(작품) 단위**로 소유되며, 해당 페이지 크리에이터만의 것이 된다.

---

## 진행 현황 (한눈에 보기)

| 단계 | 상태 | 비고 |
|------|------|------|
| **1차** | **✅ 완료** | 채팅·할일·캘린더·칸반·노트·검색 API, Socket 채팅, 마이그레이션 적용 |
| **2차** | **✅ 완료** | 설정/토글, 알림 센터, 랭킹, 멘션, 콜 UI |
| **3차** | **⬜ 미완료** | OTP/계정복구, 주소 검색, 고급 검색 |

- **✅ 완료** = 구현·마이그레이션·문서 반영까지 완료된 항목.
- **⬜ 미완료** = 아직 구현하지 않았거나, 일부만 진행된 항목.

---

## 0. 데이터 소유 원칙

| 항목 | 설명 |
|------|------|
| **단위** | 모든 신규 데이터는 `page_id`로 스코프. 한 페이지 = 한 작품(앱). |
| **소유** | 페이지는 `owner_id`(User) 소유. 해당 페이지의 채팅/할일/캘린더 등은 그 크리에이터의 앱 데이터. |
| **방문자** | 방문자는 해당 페이지에 댓글·채팅·할일 추가 등 **참여**만 가능. 데이터는 그 페이지에 속함. |
| **격리** | 페이지 A와 페이지 B의 데이터는 완전 분리. 크리에이터별로 자신만의 앱이 됨. |

## 2. 1차 상세 설계 (한 번에 구현할 범위)

### 2-1. 채팅 (실시간)

**요구**: 실시간 메시지, 페이지별 한 채팅방(또는 채널 ID 확장 가능).

| 항목 | 내용 |
|------|------|
| **DB** | `ChatMessage`: id, page_id, sender_id(optional), sender_anon_id, content, created_at. 인덱스: page_id, created_at. |
| **API** | `GET /api/pages/[pageId]/chat` (페이징, since), `POST /api/pages/[pageId]/chat` (body: content). 방문자 허용 시 anon_id 또는 user_id로 sender 식별. |
| **실시간** | Socket.IO room: `page:${pageId}`. 이벤트 `chat:message` (payload: { id, content, sender_*, created_at }). 서버에서 DB 저장 후 같은 room에 broadcast. |
| **런타임** | player.tsx: submit 시 `POST .../chat` 호출 + 기존 변수 갱신 유지. 새 메시지 목록은 기존 폴링 또는 Socket 수신으로 갱신. |
| **권한** | live 페이지만 채팅 허용할지, draft도 허용할지 정책 결정. 기본: live만. |

### 2-2. 할일 목록

| 항목 | 내용 |
|------|------|
| **DB** | `Todo`: id, page_id, title, done(boolean), sort_order(int), created_at, updated_at. 인덱스: page_id, sort_order. |
| **API** | `GET /api/pages/[pageId]/todos`, `POST` (생성), `PATCH /:id` (수정/토글), `DELETE /:id`. |
| **런타임** | 기존 패턴 매칭으로 “할일” 섹션 인식 → 해당 scope에서 GET으로 목록 로드, 체크/추가/삭제 시 API 호출. |

### 2-3. 캘린더

| 항목 | 내용 |
|------|------|
| **DB** | `CalendarEvent`: id, page_id, title, start_at, end_at(optional), all_day(boolean), meta(Json), created_at, updated_at. 인덱스: page_id, start_at. |
| **API** | `GET /api/pages/[pageId]/calendar?from=&to=` (기간 필터), `POST`, `PATCH /:id`, `DELETE /:id`. |
| **런타임** | 캘린더 섹션 인식 → 기간별 GET, 이벤트 추가/수정/삭제 시 API. |

### 2-4. 칸반 보드

| 항목 | 내용 |
|------|------|
| **DB** | `KanbanColumn`: id, page_id, title, sort_order. `KanbanCard`: id, page_id, column_id, title, body(optional), sort_order, created_at, updated_at. 인덱스: page_id, column_id, sort_order. |
| **API** | 컬럼: `GET/POST/PATCH/DELETE` .../columns. 카드: `GET` .../cards?column_id=, `POST`, `PATCH /:id` (column_id, sort_order 포함), `DELETE /:id`. |
| **런타임** | 드래그 드롭 시 PATCH로 column_id/sort_order 갱신. |

### 2-5. 노트 에디터

| 항목 | 내용 |
|------|------|
| **DB** | `Note`: id, page_id, author_anon_id(optional), author_user_id(optional), content(text), created_at, updated_at. 페이지당 1노트(싱글) 또는 여러 노트(리스트) 정책 선택. 권장: 페이지당 1노트(단일 문서). |
| **API** | `GET /api/pages/[pageId]/note`, `PUT /api/pages/[pageId]/note` (body: content). 또는 노트 목록이면 POST/PATCH/DELETE. |
| **런타임** | 노트 섹션 인식 → GET으로 로드, 저장 버튼/자동 저장 시 PUT. |

### 2-6. 검색

| 항목 | 내용 |
|------|------|
| **범위** | “페이지 내” 검색: 해당 페이지의 댓글, 채팅, 할일, 노트, 캘린더 제목 등 검색 가능 필드만 대상. |
| **방식** | PostgreSQL full-text search (to_tsvector, to_tsquery)로 충분. 별도 검색 엔진 없이 구현. |
| **API** | `GET /api/pages/[pageId]/search?q=...&type=all|comments|chat|todos|notes` (type별 테이블 검색). |
| **주소 검색** | 외부 API(카카오 주소, 네이버 등). 환경변수 `ADDRESS_SEARCH_API_KEY` 등 문서화하고, 클라이언트 또는 서버 프록시에서 호출. “좋은 방법” = API 키 있으면 외부 API, 없으면 안내 메시지. |

### 2-7. 업로드

| 항목 | 내용 |
|------|------|
| **현재** | 로컬 `public/uploads/[pageId]/...` 저장. 페이지 소유자만 업로드 가능. |
| **결정** | 현재 방식이 단일 서버/소규모에 효율적이면 **유지**. 프로덕션 확장 시를 위해 **선택 옵션**으로 S3 호환 스토리지(R2, S3, MinIO) 명세만 문서화. 환경변수 `UPLOAD_STORAGE=s3`, `S3_BUCKET`, `S3_REGION` 등. 구현은 2차에서. |

## 3. DB 스키마 추가안 (1차 — Prisma)

아래를 `prisma/schema.prisma`에 추가. 관계는 `Page`에만 걸고, 삭제 시 Cascade.

```prisma
// 채팅 (실시간)
model ChatMessage {
  id           String   @id @default(cuid())
  page_id      String
  page         Page     @relation(fields: [page_id], references: [id], onDelete: Cascade)
  sender_user_id String?
  sender_anon_id String?
  content      String
  created_at   DateTime @default(now())

  @@index([page_id, created_at])
}

// 할일
model Todo {
  id         String   @id @default(cuid())
  page_id    String
  page       Page     @relation(fields: [page_id], references: [id], onDelete: Cascade)
  title      String
  done       Boolean  @default(false)
  sort_order Int      @default(0)
  created_at DateTime @default(now())
  updated_at DateTime @updatedAt

  @@index([page_id, sort_order])
}

// 캘린더 이벤트
model CalendarEvent {
  id         String   @id @default(cuid())
  page_id    String
  page       Page     @relation(fields: [page_id], references: [id], onDelete: Cascade)
  title      String
  start_at   DateTime
  end_at     DateTime?
  all_day    Boolean  @default(false)
  meta       Json?
  created_at DateTime @default(now())
  updated_at DateTime @updatedAt

  @@index([page_id, start_at])
}

// 칸반
model KanbanColumn {
  id         String   @id @default(cuid())
  page_id    String
  page       Page     @relation(fields: [page_id], references: [id], onDelete: Cascade)
  title      String
  sort_order Int      @default(0)
  created_at DateTime @default(now())
  updated_at DateTime @updatedAt
  cards      KanbanCard[]

  @@index([page_id, sort_order])
}

model KanbanCard {
  id          String        @id @default(cuid())
  page_id     String
  page        Page          @relation(fields: [page_id], references: [id], onDelete: Cascade)
  column_id   String
  column      KanbanColumn  @relation(fields: [column_id], references: [id], onDelete: Cascade)
  title       String
  body        String?
  sort_order  Int           @default(0)
  created_at  DateTime      @default(now())
  updated_at  DateTime      @updatedAt

  @@index([page_id, column_id, sort_order])
}

// 노트 (페이지당 1개 단일 문서)
model Note {
  id              String   @id @default(cuid())
  page_id         String   @unique
  page            Page     @relation(fields: [page_id], references: [id], onDelete: Cascade)
  author_user_id  String?
  author_anon_id  String?
  content         String   @default("")
  created_at      DateTime @default(now())
  updated_at      DateTime @updatedAt

  @@index([page_id])
}
```

`Page` 모델에 relation 추가 (기존 relations 아래에 추가):

```prisma
  chat_messages    ChatMessage[]
  todos            Todo[]
  calendar_events  CalendarEvent[]
  kanban_columns   KanbanColumn[]
  note             Note?
```

업로드 정책: 현재는 페이지 소유자만 업로드 가능. 방문자 업로드(채팅 첨부, 폼 제출 등)가 필요하면 2차에서 `POST /api/pages/[pageId]/upload`를 live 페이지에 한해 anon_id 허용하도록 확장 가능.

## 4. API 라우트 목록 (1차)

| 메서드 | 경로 | 설명 |
|--------|------|------|
| GET | `/api/pages/[pageId]/chat` | 메시지 목록 (query: limit, since) |

| GET | `/api/pages/[pageId]/calendar` | 이벤트 목록 (query: from, to) |
| POST | `/api/pages/[pageId]/calendar` | 이벤트 생성 |
| PATCH | `/api/pages/[pageId]/calendar/[id]` | 수정 |
| DELETE | `/api/pages/[pageId]/calendar/[id]` | 삭제 |
| GET | `/api/pages/[pageId]/kanban/columns` | 컬럼 목록 |
| POST | `/api/pages/[pageId]/kanban/columns` | 컬럼 생성 |
| PATCH | `/api/pages/[pageId]/kanban/columns/[id]` | 컬럼 수정/순서 |
| DELETE | `/api/pages/[pageId]/kanban/columns/[id]` | 컬럼 삭제 |
| GET | `/api/pages/[pageId]/kanban/cards` | 카드 목록 (query: column_id?) |
| POST | `/api/pages/[pageId]/kanban/cards` | 카드 생성 |
| PATCH | `/api/pages/[pageId]/kanban/cards/[id]` | 카드 수정/이동 |
| DELETE | `/api/pages/[pageId]/kanban/cards/[id]` | 카드 삭제 |
| GET | `/api/pages/[pageId]/note` | 노트 1개 |

## 5. 런타임 연동 포인트 (player.tsx)

| 자산/패턴 | 현재 | 변경 |
|-----------|------|------|
| 채팅룸 submit | mock 또는 변수만 갱신 | POST `/api/pages/{pageId}/chat` 호출, 성공 시 Socket으로 수신된 메시지 반영 또는 목록 재요청 |
| 할일 | 변수/로컬 상태 | GET todos 로드, 체크/추가/삭제 시 PATCH/POST/DELETE |
| 캘린더 | 변수/로컬 | GET calendar?from=&to=, 이벤트 CRUD API |
| 칸반 | 로컬 | GET columns/cards, 드래그 시 PATCH card (column_id, sort_order) |
| 노트 | 로컬 | GET note, 저장 시 PUT note |
| 검색 | 클라이언트 필터 | GET search?q=... 호출 후 결과 표시 |
| 업로드 | 현재 `/api/app/{pageId}/upload` | 유지. 필요 시 나중에 S3 연동 |

프리셋 버튼/입력은 이미 `wireSubmit`/`wireMockSubmit`으로 연결 가능. 실기능 전환 시 해당 프리셋의 submit URL을 위 API로 바꾸고, player에서 payload 필드 매핑(예: content, title, done)만 맞추면 됨.

## 6. 작업 순서 (1차 한 번에 처리 권장)

1. **스키마**: Prisma 모델 추가 → `npx prisma migrate dev --name add_asset_features`  
2. **API**: 위 라우트 전부 구현 (인증: anon 또는 page live 여부만 체크).  
3. **Socket.IO**: 채팅 메시지 저장 후 room broadcast.  
4. **런타임**: player.tsx에서 채팅/할일/캘린더/칸반/노트/검색 패턴에 대해 위 API 호출 및 상태 반영.  
5. **프리셋**: 필요 시 assetLibraryPresets에서 wireSubmit URL을 실제 엔드포인트로 변경.  
6. **검색**: PostgreSQL full-text (comments, chat_messages, todos, note 등) + GET search.  
7. **업로드**: 코드는 유지, docs에만 S3 옵션 명세 추가.

## 7. 필요한 환경/외부 요약

| 항목 | 필요 여부 | 비고 |
|------|-----------|------|
| DB | 필수 | 기존 PostgreSQL 유지. 마이그레이션만 추가. |
| Socket.IO | 필수 | 이미 사용 중. 채팅 이벤트만 추가. |
| 이메일 (OTP/복구) | 2차 이후 | SendGrid/Resend 등 API 키. |
| 주소 검색 API | 선택 | 카카오/네이버 등. 키 있으면 연동, 없으면 안내. |
| S3/R2 업로드 | 선택 | 프로덕션 확장 시. 아래 7-1 참고. |

### 7-1. 업로드 스토리지 옵션 (S3 호환)

현재 `POST /api/app/[pageId]/upload`는 로컬 디스크 `public/uploads/[pageId]/`에 저장. 단일 서버·소규모에 적합.

**프로덕션 확장 시** (선택): S3 호환 스토리지로 전환 가능.

| 환경변수 | 설명 | 예시 |
|----------|------|------|
| `UPLOAD_STORAGE` | `local`(기본) 또는 `s3` | `s3` |
| `S3_BUCKET` | 버킷 이름 | `my-app-uploads` |
| `S3_REGION` | 리전 | `ap-northeast-2` |
| `S3_ACCESS_KEY_ID` | IAM 액세스 키 | (보안에 따라 환경만 주입) |
| `S3_SECRET_ACCESS_KEY` | IAM 시크릿 | (보안에 따라 환경만 주입) |
| `S3_ENDPOINT` | (선택) R2/MinIO 등 커스텀 엔드포인트 | `https://xxx.r2.cloudflarestorage.com` |

구현 시: `UPLOAD_STORAGE=s3`일 때만 S3 클라이언트 사용, 업로드 후 `url`을 버킷 공개 URL 또는 서명 URL로 반환. 미설정 시 기존 로컬 저장 유지.

### 9-4. 정리

- **실기능으로 동작하는 것**: 채팅(전송·목록 갱신), 댓글, 신고, upvote, 업로드, 1·2차 API(할일/캘린더/칸반/노트/검색/설정/알림/랭킹/멘션/콜 상태). 배포·미리보기 시 `appPageId`가 있으면 해당 페이지로 요청.
- **시뮬레이션/UI만**: 결제, OTP, 주소 검색, 일부 차트·대시보드 등 외부 API 또는 미연동 UI. 3차 항목은 API 키 제공 시 연동 가능.

### 9-5. 프리셋 약 120개 — 실제 작동·검사 가능 여부

| 구분 | 개수(대략) | 설명 |
|------|------------|------|
| **실기능 + 검사 가능** | **확대** | 채팅룸·라이브 채팅, 댓글, 신고, upvote, 업로드(3곳), **할일(추가 버튼)**, **노트(저장 버튼)** — `wireSubmit` + 런타임 초기 fetch로 연동. |
| **데이터 로드 실기능** | **9개** | 할일·캘린더·칸반·노트·설정·알림·랭킹·칸반 — 페이지 로드 시 `appPageId` 기준으로 API 자동 호출 후 목록/내용 채움. 추가/저장 버튼은 할일·노트만 실제 전송, 나머지는 UI만. |
| **UI·흐름만 검사 가능** | **대부분** | 헤더/탭바/사이드바, 모달, 폼 검증, 스켈레톤, 에러 페이지, 결제·OTP·주소검색 등 — 시뮬레이션 또는 미구현. |

- **적용 완료 (120개 실기능 확대)**  
  - **런타임 초기 fetch**: `appPageId` 있을 때 todos, calendar, note, settings, notifications, ranking, kanban 한 번에 GET 후 변수 오버라이드로 목록/내용 채움.  
  - **제출 후 refetch**: 채팅 외에 `/todos`, `/note`, `/calendar` 제출 성공 시 해당 API 재요청 후 화면 갱신.  
  - **프리셋 연동**: 할일 — "새 할일" 입력 + "추가" → `POST .../todos`; 노트 — "내용" 입력 + "저장" → `PUT .../note`.  
- **결론**: "120개 전부"는 **데이터 로드까지 포함해 9개 영역(할일·캘린더·칸반·노트·설정·알림·랭킹·칸반 등)은 실데이터 표시**, 그중 **할일·노트는 추가/저장까지 실기능** 동작. 나머지 프리셋은 UI/흐름 검사 또는 기존 wireSubmit(채팅·댓글·신고·업로드 등)으로 검사 가능.

### 9-6. 검증 방법 (실기능 연동된 프리셋)

1. **앱 실행**  
   `npm run dev` (기본 3000 포트. 3100 쓰려면 `PORT=3100 npm run dev` 또는 package.json 스크립트 수정.)

2. **페이지 열기**  
   - 로그인 후 **작품(페이지) 하나** 생성 또는 기존 페이지 선택.  
   - 주소: `http://localhost:3000/editor/advanced?pageId=<페이지ID>`

3. **실기능 프리셋 넣고 미리보기**  
   - 자산 라이브러리에서 **채팅룸** 또는 **라이브 채팅** 프리셋 삽입.  
   - 상단 **미리보기** 클릭.

4. **동작 확인**  
   - 채팅 입력란에 글 쓰고 **전송** 버튼 클릭.  
   - 브라우저 **개발자 도구 → Network**: `POST /api/pages/<pageId>/chat` 요청이 가고 200 응답 오는지 확인.  
   - 같은 방식으로 **댓글 스레드** 프리셋은 댓글 작성/수정/삭제, **콘텐츠 리포트**는 신고 버튼, **추천** 프리셋은 추천/취소 버튼으로 검증.

5. **API만 직접 검증** (선택)  
   - 터미널에서 `curl` 또는 Postman으로 예:  
     - `GET http://localhost:3000/api/pages/<pageId>/chat`  
     - `POST http://localhost:3000/api/pages/<pageId>/chat` (body: `{"content":"테스트"}`)  
   - 익명 사용자면 `x-anon-user-id` 헤더 필요(먼저 `POST /api/anon/init` 호출해 anon_user_id 획득).

6. **3100 포트 종료**  
   - Windows: `netstat -ano | findstr :3100` → 나온 PID로 `taskkill /PID <PID> /F`  
   - 또는 해당 터미널에서 서버 실행 중이면 `Ctrl+C`
