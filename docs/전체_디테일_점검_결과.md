## 2. API·라우트 점검

### 2.1 인증·권한

| 항목 | 상태 | 비고 |
|------|------|------|
| login/signup/anon/init rate limit | ✅ | `lib/rate-limit.ts` + LOGIN/SIGNUP/ANON_INIT_MAX_PER_MINUTE |
| 세그먼트 GET/POST/PATCH/DELETE | ✅ | page 소유자(anon_id) 검증 후 raw SQL |
| analytics 전반 | ✅ | resolveAnonUserId 후 owner 또는 live/deployed 시에만 허용 |
| events/ingest | ✅ | pageId만 검증(공개 수집), 인증 없음·CORS `*` |

### 2.2 §31·§32 관련 API

| API | rate limit | 인증 | 스펙 대비 |
|-----|------------|------|-----------|
| `GET/PATCH .../alerts/settings` | 없음 | owner | discord_webhook_url·scheduled_report_enabled·auto_drop_alert raw SQL ✅ |
| `POST .../alerts/notify`, `test` | 5회/분 | owner | ✅ |
| `GET .../analytics/export` | 10회/분 | owner | ✅ |
| `GET .../analytics/health` | 없음 | owner | last_event_at, visits_today/yesterday, drop_warning, **gaps** ✅ |
| `GET .../analytics/engagement` | 없음 | owner | by_day, sessions_over_30s ✅ |
| `GET .../heatmap` | 30회/분 | owner | compare=1 시 previous_grid ✅ |
| `GET .../events`, `events/export` | 30회/분, 10회/분 | owner | ✅ |
| `POST .../events/ingest` | 60회/분 | 없음(page만) | CORS, synthetic LiveSession + Event ✅ |
| `GET/POST/PATCH/DELETE .../segments` | 없음 | owner | 전부 raw SQL, page_id 소유 검증 ✅ |

### 2.3 SQL·보안

- **세그먼트**: INSERT/UPDATE/DELETE/SELECT 모두 `$queryRaw`/`$executeRawUnsafe` 사용. PATCH 시 `WHERE id = $${idx + 1}`로 바인드 올바름.
- **alerts/settings**: discord_webhook_url 등 조회/갱신 시 raw SQL, 파라미터 바인드 사용.
- **일반**: Prisma 사용 구간은 파라미터화됨. SQL 인젝션 방지 적절.

### 2.4 엣지·에러 처리

- **ingest**: `body.events` 없거나 빈 배열 → 200 + `received: 0`. page 없음 → 404. 잘못된 JSON → 400.
- **health**: period 비 7d/30d → 400. page 없음/삭제/비소유 → 404/403.
- **segments PATCH**: 수정 필드 없으면 200 + 기존 segment 반환. segmentId가 해당 page 소유 아님 → 404.

## 3. DB·스키마

- **Segment**: raw SQL 전용. Prisma generate 없어도 segments API 동작.
- **DailyPageStats**: `prisma.dailyPageStats.upsert` 사용(cron). `@@unique([page_id, date])` → `page_id_date` 복합 조건. **Prisma generate 필요.** generate 미실행 시 해당 블록은 catch로 무시되어 장애 없음.
- **Event**: live_session_id, ts 인덱스 등 스키마 일치.
- **마이그레이션**: Segment, DailyPageStats 모델 반영 여부는 `prisma migrate status`로 확인 권장.

## 6. 발견 이슈·권장 사항

### 6.1 경미·선택

1. **engagement 대시보드 노출**  
   - `GET .../analytics/engagement`는 구현됨. 대시보드에 "체류 30초 이상 세션" 또는 engagement 전용 작은 블록을 두면 §31.5 코호트·리텐션 설명과 더 잘 맞음(선택).

2. **ingest 대량 이벤트**  
   - 요청당 최대 100건, Event.create를 루프로 수행. 매우 큰 트래픽 시 createMany 또는 트랜잭션 배치 검토 가능(현재 스펙 범위 내).

3. **CORS (ingest)**  
   - `Access-Control-Allow-Origin: *`. 운영에서 도메인 제한이 필요하면 환경변수 등으로 제한 검토.

### 6.2 운영·배포

4. **Prisma generate**  
   - DailyPageStats 사용 cron은 generate 후에만 upsert 동작. 배포 파이프라인에 `prisma generate` 포함 권장.

5. **Cron 설정**  
   - 예약 리포트·자동 급감 알림: 스케줄러에서 `GET /api/cron/daily-reports` 매일 호출, `Authorization: Bearer <CRON_SECRET>` 필수.

6. **rate limit 헤더**  
   - login/signup/anon/init/ghost POST에 X-RateLimit-* 등 반환됨. export/heatmap/events 등 §31 관련 API는 429 시 메시지만 있고 헤더는 선택(일관성 위해 추가 가능).

## 8. Prisma 실행 안내

- **스키마**: `prisma/schema.prisma`에 Segment, DailyPageStats 등 §31·§32용 모델 반영됨.
- **마이그레이션**: `prisma/migrations`에 `20260208100000_daily_page_stats`, `20260209000000_add_segment` 등 적용된 상태.
- **로컬에서 할 일**  
  - **Client 생성**: 터미널에서 `npm run prisma:generate` 또는 `npx prisma generate` 실행.  
  - DailyPageStats(cron)·Segment(선택)를 Prisma 클라이언트에서 쓰려면 generate 필수.
- **EPERM(operation not permitted)** 가 나면: dev 서버·다른 Node/Prisma 프로세스를 끈 뒤, 프로젝트 루트에서 다시 `npx prisma generate` 실행.
- **DB 마이그레이션 적용**: 로컬 DB에 마이그레이션을 아직 안 넣었다면 `npm run db:migrate` 또는 `npx prisma migrate dev` 실행.
