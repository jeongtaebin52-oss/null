# NULL — 노코드 풀스택 로드맵 (Figma + Wix 상위호환)

> **목표**: 진짜 백엔드·프론트엔드 개발 수준을 노코딩으로 구현. Figma + Wix 상위호환.  
> **기준**: 각 항목을 완료하면 해당 항목 제목에 **✅ 완료** 표시. 순서대로 진행 권장.  
> **현재**: EXECUTION_ORDER 1~31까지 에디터·런타임·인프라 완료. 본 문서는 그 위에 “노코드 풀스택”을 쌓는 단계.

---

## 원칙: 작품(프로젝트) 단위 데이터 격리

- **공용이 아님.** NULL에서 만드는 건 “진짜 실사용 앱/웹”이므로, **각 사용자의 각 작품(프로젝트)마다 그 작품만의 데이터**를 써야 한다.
- 즉, **작품 A의 DB/테이블/API**와 **작품 B의 DB/테이블/API**는 완전히 분리. 한 작품의 데이터·스키마가 다른 작품에 노출되거나 섞이면 안 됨.
- **구현 방향 (둘 중 선택 또는 혼합)**  
  - **논리적 격리**: 하나의 물리 DB를 쓰되, 모든 앱 데이터 테이블에 `page_id`(또는 `project_id`)를 두고, CRUD·API는 “현재 작품 ID”로만 조회/수정. 스키마(모델 정의)는 작품별로 JSON/메타로 저장.  
  - **물리적 격리**: 작품별로 별도 DB(예: SQLite 파일) 또는 별도 스키마(예: PostgreSQL schema per project). 격리·보안은 최대지만 운영·마이그레이션 복잡도 증가.
- 아래 1~11 항목은 **전부 “현재 편집/실행 중인 그 작품” 기준**으로만 데이터·API·권한이 동작한다고 보면 됨.

---

## 1. 데이터 모델 에디터 (백엔드 스키마 정의)

- **작품 단위**: 이 작품(페이지/프로젝트) 전용 데이터 모델만 정의·저장. 다른 작품 모델과 공유하지 않음.
- 에디터 또는 전용 화면에서 **데이터 모델** 정의: 테이블(컬렉션) 추가/삭제, 필드(이름·타입: string/number/boolean/date/json/relation) 추가/삭제.
- 스키마는 “이 작품” 메타(예: `page_id` + JSON 스키마)로 저장. Prisma 동적 스키마 또는 JSON 기반 런타임 쿼리 등으로 구현.
- **완료 시**: 본 항목 제목에 `✅ 완료` 추가.

## 2. CRUD API 자동 생성

- **작품 단위**: 요청 시 “현재 작품(page_id/project_id)”이 반드시 포함. 해당 작품의 모델·데이터만 접근.
- 데이터 모델(해당 작품 기준) 기준으로 **목록(GET)·단일 조회(GET :id)·생성(POST)·수정(PATCH/PUT)·삭제(DELETE)** API 자동 생성.
- 라우트: 예 `GET/POST /api/app/[pageId]/[model]`, `GET/PATCH/DELETE /api/app/[pageId]/[model]/[id]` 처럼 **작품 ID가 경로에 포함**. 인증/권한은 4번에서.
- **완료 시**: 본 항목 제목에 `✅ 완료` 추가.

## 3. 리스트·테이블 노드 + 데이터 바인딩 (프론트)

- **리스트/테이블 노드** 추가: “데이터 소스”에 이 작품의 API URL 또는 모델명 지정(자동으로 현재 작품 context 포함), 응답 배열을 행으로 반복 렌더. 클릭 시 상세 페이지로 이동(선택 행 ID 전달).
- **데이터 바인딩**: 텍스트/이미지 노드에 `{{ 변수.필드 }}` 또는 `{{ 리스트행.필드 }}` 형태로 바인딩. submit 응답·API 조회 결과는 **해당 작품의 데이터만** 반영.
- **완료 시**: 본 항목 제목에 `✅ 완료` 추가.

## 4. 페이지·API 접근 권한 (인증/역할)

- **페이지 접근 조건**: “이 페이지는 로그인 사용자만” / “특정 플랜만” 등 설정. 런타임에서 조건 미충족 시 로그인 페이지 또는 에러 페이지로 리다이렉트.
- **API 접근 권한**: CRUD API별 “역할/플랜별 허용” 규칙 설정. (예: 본인 데이터만 수정 등.)
- **완료 시**: 본 항목 제목에 `✅ 완료` 추가.

## 6. URL ↔ 페이지/변수 (라우팅·딥링크)

- **URL 규칙**: 예 `/p/[pageId]`, `/product/[id]` 등. 경로/쿼리 파라미터를 “페이지 ID + 변수(예: productId)”로 매핑.
- 런타임: URL 읽어서 `initialPageId` 및 변수 세팅. 링크 공유 시 해당 화면·상태로 진입.

## 7. 파일 업로드 컨트롤

- **작품 단위**: 업로드 저장 경로/버킷은 **현재 작품(page_id)** 기준으로 격리. 다른 작품 파일과 혼용되지 않음.
- **파일 업로드 노드**: 이미지/문서 선택 → 업로드 API 호출(현재 작품 context 포함) → 반환 URL을 변수에 저장. 폼 submit 시 해당 변수 포함.
- **완료 시**: 본 항목 제목에 `✅ 완료` 추가.

## 11. 최종 점검 — 노코드 풀스택 완성도

- 위 1~10 항목 대조. “**작품 단위 데이터 격리** + 데이터 모델 → CRUD API → 화면 바인딩 → 권한 → 폼·라우팅·파일·컴포넌트·로직·배포”가 노코드로 가능한지 확인.
